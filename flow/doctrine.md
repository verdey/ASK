# Flow — Doctrine

> *Filesystem telepathy. Every LOB self-narrates from a loud `init.md` into a serialized process tree, and graduates by degrees from Claude-driven markdown → script → app. Living documentation as reference. Acid test: whatever `ls` shows in ascending order IS the order of the Flow.*
> — Dan Greeney, 2026-04-26

---

## What a "flow" is

A **flow** is the motion of a Line of Business (LOB). It has four parts:

1. **A loud entry point** — `init.md` at the LOB root. Lowercase, wu wei, alphabetically first. Names the mission, the cold-boot read order, the current stage, the graduation tier, and a `Blueprint reference` footer that pins the last sync against the living blueprint. Paired with `index.html` — the flow's **living visual report** (see *The living report* section below).

2. **A serialized process tree** — `processes/` with subdirs in OS-ascending-sort order. Each subdir is a stage. The order the filesystem shows IS the order of the work. This is **The Acid Test**: no separate index, no manifest, no scheme — `ls` is the source of truth.

3. **A graduation ladder** — every stage carries a tier (0 through 3) and the upgrade path is named in advance. Tier 0 is markdown Claude runs by hand; Tier 1 is a CLI script; Tier 2 is a PWA/HUD/function for roster self-serve; Tier 3 is an integrated app. A stage advances only when **volume bark + audience bark + stakes bark** fire simultaneously. Until then, the file version is enough.

4. **A living blueprint** — `_flow-blueprint/` at portfolio root. Itself a flow. Active flows clone from it, reference it on cold-boot, and re-sync against it as it improves through AARs. Not a template (copy-and-discard) — a blueprint (consult repeatedly).

## The living report

Every flow must produce `index.html` at its root. This file is the flow's **alpha and omega**:

- **Alpha** — the cold-boot entry point. Anyone landing on the flow reads `index.html` first: what is this flow, what does it do, where does it stand?
- **Omega** — the final word from the run. Because the render step (`0600-render-index-html/`) runs LAST in every flow's `processes/` tree, it sees every preceding step's outputs. It reports on what the flow produced, what patterns emerged, what the health signal is.

`index.html` is not a static render of `init.md`. It is the flow's **living visual report**. It also gets surfaced upward by stitch flows (`flows_deals/`, roster dashboards) — the richer the omega report, the more useful the upstream stitching.

**Required sections** (generated by the `0600-render-index-html/` step in the blueprint):

| Section | Source | Purpose |
|---------|--------|---------|
| **Process map** | `ls processes/` | Visual step cards: name, tier badge (▰▱▱▱ → ▰▰▰▰), last-run timestamp, exit status (✓ / ✗ / —) |
| **Recent activity** | `_audit/runs.jsonl` | Last 10 run entries; degrades to "No runs yet." |
| **AI insights** | LLM-authored at render time | 3–7 curated observations: patterns, bottlenecks, growth areas, health signal |
| **Blueprint sync** | `init.md` Blueprint reference + `_flow-blueprint/CHANGELOG.md` | Last-synced date + unabsorbed CHANGELOG entries (✓ or ⚠ list) |
| **init.md body** | `init.md` | Existing embedded-md-viewer render; unchanged |
| **Backlog / suggestions** | `docs/backlog/` or `_suggestions/` | Optional; omit if empty |

**Override pattern:** Flow inits may declare overrides or expansions in `init.md` under a `## Report customizations` section. The four required sections above are the floor — flows can add, not subtract.

**Exemplar richness targets:**
- `Finance/Income/flow-navigator.html` — canonical target: Next 3 Moves, Brief from Claude, 7-day cadence heartbeat, Decisions Queue, Holographic LOB grid, Bark conditions
- `Live/flows_deals/index.html` — stitch archetype: filterable accordion, broken-asset defanging, source citations
- `Finance/Income/Flows/Roster/*/index.html` — persona-filtered composition variant

**Positioning rule:** The render step must always be the **highest-numbered step** in `processes/`. Whatever number the flow uses (the blueprint uses `0600`), it must sort last under `ls`. The Acid Test is the enforcer: if `ls processes/` shows the render step anywhere but last, that is a Filesystem-Truth violation.

**Regeneration trigger:** On every flow run after all preceding steps complete; also on demand when `processes/` structure or `init.md` changes.

**Under-specification bark:** If `index.html` exists but contains only the init.md body (no process map, no activity, no insights, no sync), it is under-specified — flag for regeneration via the `0600-render-index-html/` step.

## The three axioms

**The Acid Test.** *Whatever `ls` shows in ascending order IS the order of the Flow.* No exceptions. If the filesystem and the doctrine disagree, the filesystem wins — or the doctrine gets amended to match.

**Filesystem-Truth.** *Every name or reference in a flow's tree must agree with what is actually present at that path.* Drift between claim and content is the highest-priority fix. Either build the missing thing or amend the claim. Never both, never neither. Papering over drift IS the violation.

**Runtime declaration is mandatory.** *Every step must declare its runtime: `LLM`, `SCRIPT`, or `composite` in its `instructions.md` or `step.md`.* The declaration removes ambiguity about whether a step is in-LLM-budget and gates execution against the runner's capability matrix. A step without runtime declaration is under-specified, even if dry-run passes. Precedent: step 0400-author-omega caused a silent failure (no omega written despite run-complete toast) because SCRIPT-tagged steps had no executor in the runner version in use; dry-run passed because it validates shape, not runtime implementation. The declaration is the pre-flight gate. (Source: `Tooling/flow-omega-author/docs/sessions/_S1-heal-the-runtime-gap.md` §7 L2; promoted 2026-04-28 via `/flow curate-batch` W3 cassiopeia.)

**Audit corollary — documented intent over shape inference.** Before flagging a step or folder as orphan, drift, or misplaced: read its `step.md` / `README.md` / `_graduation.md`, check `docs/decisions/` for an ADR that justifies it, and re-read the parent flow's `init.md` for archetype-with-variant declarations. A step whose declared intent contradicts a recommendation downgrades that recommendation to "verify intent" before any subtract/merge. Documented intent beats archetype assumption — every time. Precedent: 2026-04-27 audit of `_flow-navigator` would have deleted the Phase 3 co-creation harvest step as "orphan tail" without this rule (`lessons.md` 2026-04-27).

**Audit corollary — runtime declaration vs. executor support (added 2026-04-28, carmen.sheet-bend, per lessons.md L4).** When a step declares `**Runtime:** SCRIPT` or `**Runtime:** composite` in its `instructions.md`, audit must verify two things: (1) the step has a `## Invocation` section with a fenced bash block (for SCRIPT), and (2) the runner version in use supports the declared runtime — check `Tooling/flow-runner-llm/CLAUDE.md` §Runtimes for the current support matrix. If the runner lacks the executor, surface as ⚠️ `runtime-not-implemented`: the flow is *structurally correct but functionally non-executable* via the portal trigger surface. A dry-run green is not a live-click green. Precedent: `flow-omega-author` passed dry-run with `Runtime: SCRIPT` steps but produced no omega on live click until the dispatcher landed (commit `80f1862`, carmen.overhand); MHTML capture then confirmed functional end-to-end (commit `88bb3b8`, carmen.square). `runtime-not-implemented` flows should surface the `no omega` chip rather than a spurious run-complete toast.

## What a flow is NOT

- **Not a template.** A template is copy-and-discard. A flow's blueprint is a living reference that flows look back at.
- **Not a kanban board.** Stages are not "to do / doing / done." They are *the work itself*, in order.
- **Not a database schema.** Files are the API. Markdown is the schema. Filesystem is the database. (per `thin-engine.md`)
- **Not a CI/CD pipeline.** A flow may eventually graduate to one. Most never need to.
- **Not a checklist.** A checklist is steps for a person; a flow is the LOB's motion, with its own state, artifacts, and graduation pressure.

## The step-folder canon (settled) and step-content shapes (open)

**Settled.** The Acid Test fixes the *folder* convention with no ambiguity: every step is a folder under `processes/` with a numeric prefix, OS-ascending-sortable. `0010-discover/` before `0020-curate-registry/` before `0030-stitch/`. Whatever `ls` shows IS the order. Subflows nest the same way. This is non-negotiable.

**Open.** The *content shape inside each step folder* is intentionally pluralistic right now. Three valid shapes coexist:

1. **Skinny artifact** — one primary `.md` with YAML frontmatter (`flow`/`role`/`upstream`/`downstream`/`status`) + body + `## Refine`. No README, no `_graduation.md`. The artifact IS the step.
2. **Legacy pair** — `README.md` (the executable prompt/logic) + `_graduation.md` (the testable next-tier criteria). Heavier, but earns its weight once a step has graduated past Tier 0 to a real script/PWA/app.
3. **Prose-skinny** — a single `step.md` with H2 sections (`## Purpose / Inputs / Outputs / Run / Refine`) and no YAML frontmatter. Born of the `flows_deals/` build (April 2026); easier to author, harder to lint, sufficient for everything observed so far.

**Why we don't pick a winner yet.** We don't have enough data on which shape fails which way at scale. Forcing a canonical choice now would lock in whichever shape happened to be most recently authored. Instead, we let all three live and gather evidence.

## Step self-logging (the data we need to settle the shape question)

Every step run, regardless of shape, emits a single JSONL line to `<flow-root>/_audit/runs.jsonl`:

```json
{"ts":"2026-04-27T22:14:03Z","flow":"flows_deals","step":"0030-stitch","shape":"prose-skinny","surface":"mind","duration_s":1,"exit":0,"manual_fixups":0,"notes":""}
```

| Field | What it captures | Why it matters |
|-------|-----------------|----------------|
| `ts` | UTC timestamp | When |
| `flow` | flow slug (parent of `processes/`) | Which flow |
| `step` | step folder basename | Which step |
| `shape` | `skinny` / `legacy` / `prose-skinny` / `script-only` | The shape question, tagged at run time |
| `surface` | `mind` / `hands` / `auto` / `human` | Who/what ran it |
| `duration_s` | wall time | The pain proxy |
| `exit` | 0 / non-zero | Did it work |
| `manual_fixups` | int — operator deviations from the script | The friction signal |
| `notes` | optional one-liner | Anything notable |

The `0099-run-all` orchestrator (Stitch archetype) appends one line per step. Hand-run Tier-0 steps append via a small helper (`bin/log-run.sh` — to be canonized in the blueprint). After ~50 runs across enough flows, Streamline can reason about: which shapes correlate with high `manual_fixups`, which steps stall most often, which surfaces (Mind vs Hands vs auto) fit which shape best. **The data picks the canonical shape, not the doctrine.**

Until then, Audit treats all three shapes as valid and only flags **absence of any shape** — a step folder with no `step.md`, no `README.md`, and no skinny artifact is the violation. Self-logging is the only requirement that's universal.

## Flow archetypes

A flow's *archetype* answers "what shape of work does it do?" Three are recognized so far:

### Workflow flow
The pipeline pattern from `workflow-lob.md`. Steps consume upstream inputs, produce downstream outputs, terminate in a deliverable. Examples: `Income-Land-Deal-Search` (upstream sources → harvest → notify → dashboard), `wholesaling/intake-funnel`. The flow IS the deliverable's production line.

**Variant: main-pipeline-as-flowchart + SubFlow-sidecar.** When a Workflow LOB's main pipeline is *operational doctrine* (humans-and-SOPs, not an automated chain), the canonical flow may live as a Mermaid flowchart in `init.md` plus codifying SOPs in `docs/sop/`, while `processes/` holds **out-of-band SubFlows** that read from / write to the LOB but don't gate it. SubFlows follow `processes/<roster-user>-<purpose>/` (e.g., `intake-funnel/`, `amber-top-of-mind/`). The Acid Test still applies *within each SubFlow*; the main pipeline's sequence is carried by the flowchart + SOP set, not by `ls processes/`. Example: [`wholesaling`](file:///Users/verdey/Documents/Claude/Projects/Finance/Income/Flows/LOBs/wholesaling/) — lead-capture → qualify → walk → offer → close lives in init.md flowchart; `processes/` holds SubFlows that emit fragments for roster digest pages. Audit must classify this variant as "Workflow / SubFlow-sidecar," not as "main pipeline missing."

**Variant: workflow with co-creation write-back tail.** A Workflow flow may extend its terminal `notify` step into a feedback loop: the deliverable embeds an inbox-write surface (Dan picks options inline), a downstream `harvest-inbox` step reads decision stanzas and applies them across the portfolio, and `notify` documents where push-style alerts land when state changes meaningfully. The tail is intentional, not orphan. Example: [`_flow-navigator`](file:///Users/verdey/Documents/Claude/Projects/Finance/Income/Flows/_flow-navigator/) — `05-harvest-inbox/` reads Dan's stanzas from `Income/docs/flow-navigator/inbox/` and rewrites SOPs/ADRs across LOBs (see ADR `0002-co-creation-write-back.md`).

### Stitch flow
A read-only-over-substrate flow whose deliverable is **one stable artifact at a known path** that aggregates outputs from elsewhere. Discovers candidates → curates by registry → stitches into a unified surface → logs the run. Examples: [`Income-Flow-Navigator`](file:///Users/verdey/Documents/Claude/Projects/Finance/Income/Flows/_flow-navigator/) (holographs the portfolio), [`flows_deals`](file:///Users/verdey/Documents/Claude/Projects/Live/flows_deals/) (accordions every flow's HTML outputs).

The Stitch contract:
1. **Stable artifact path** (`dist/index.html` or equivalent) — overwritten in place, never moves.
2. **Dated snapshots beside it** (`dist/snapshots/<TS>/index.html`) — for diff/comparison.
3. **Self-refresh comment block at top of artifact** — any Claude surface can read it and re-run the pipeline; no out-of-band runbook.
4. **Read-only over substrate** — the flow never writes into the trees it reads from. The constraint is the feature.
5. **Defang, don't break** — when stitching foreign content (other flows' HTMLs, other LOBs' artifacts), broken references are the default, not the exception. Degrade gracefully (sentinel attrs + ⚠ chips + count) rather than aborting.
6. **Declarative knobs separate from mechanical execution** — a step that has both judgment (which kinds, which retention, which ordering) and transformation (the actual work) splits into `<step>/<rules>.yaml` + `<step>/<exec>.{sh,py}`. Streamline can leave the YAML alone.
7. **Live HTTP surface** (Herd valet domain, e.g. `flows.test`) — symlink the realm root or `dist/` so the verification loop is `curl https://flows.test/`, not `open file://…`.

### Catalog flow
A specialized workflow archetype for converting oversized static prose documents (500+ lines with internal structure) into self-contained flow entries. Each entry is independent; there are no sequential dependencies or graduation gates. Users select one entry to act on, not steps through a pipeline.

**Numbering convention for catalogs:** 2-digit zero-padded (`01-`, `02-`, …, `NN-`) for small fixed-count collections (recommended up to ~50 entries). Preserves source document ordering while keeping `ls processes/` output scannable. For larger or unbounded catalogs, fall back to 4-digit. **Distinction from pipeline flows:** pipelines use 4-digit (`0100-`, `0200-`) to reserve insertion space between stages; catalogs use 2-digit because the entry count is known and final.

**Catalog `init.md` variant:** Instead of describing a sequential pipeline, catalog inits present a **selection flow** — metadata table of available entries, selection criteria or guidance, graduation tier, and blueprint reference. No Mermaid flowchart of main-pipeline gates. The acid test applies within each entry's step.md.

**Example:** `Library/tome-of-geometry/flows/prototype-patterns/` — 25 independent geometry/design pattern entries (01-Euclidean through 25-Applied), each a self-contained build target. Sourced from wave-2 absorption (K-EA2).

### Living-blueprint flow
A meta-flow that is itself a flow AND the canonical reference other flows clone from. The root instance is `_flow-blueprint/` at the kingdom root; intermediate blueprints (e.g., a "short-sale-deal blueprint" scoped to one LOB) are also Living-blueprint flows that nominate `_flow-blueprint` as their own parent. Maintains a `CHANGELOG.md`; active flows pin a `Last synced` date per ancestor and re-sync on cold-boot when any ancestor advances.

## Composition (parent/child)

Composition is **orthogonal to archetype.** A flow can be Workflow-shaped and also be a subflow of another Workflow, or a Catalog inside a Stitch, etc. The archetype dimension answers "what shape of work?"; the composition dimension answers "what's its relationship to other flows?"

**Three structural shapes,** all declared via YAML frontmatter at the top of `init.md`:

```yaml
---
flow_id: <kebab-slug>
archetype: workflow | stitch | catalog | living-blueprint
parent_flows:
  - <path-or-id>
child_flows:
  - <path>
blueprint_lineage:
  - <root-blueprint>
  - <intermediate-blueprint>
---
```

1. **Subflow.** A numbered step folder under `processes/` that contains its own `init.md` + `processes/`. The Acid Test holds at every level — each `processes/` is independently `ls`-ordered. Recursion stops at folders without `processes/` (those are leaf steps with `README.md` / skinny YAML / `step.md`). Existing example: `wholesaling/processes/intake-funnel/` is a subflow of `wholesaling/`.
2. **Multi-level blueprint inheritance.** A flow can declare a chain of ancestor blueprints in `blueprint_lineage`. Example: a "short-sale deal blueprint" inherits from `_flow-blueprint`; individual deals inherit from the short-sale blueprint. Cold-boot walks the chain root-first and surfaces CHANGELOG divergence per ancestor. The denormalized chain saves the cold-boot from re-traversing.
3. **Instance flow.** A one-off flow (a deal, a project, a campaign) that points at an intermediate blueprint via `parent_flows`. Improvements to the blueprint surface to the instance via the same CHANGELOG-divergence mechanic that propagates `_flow-blueprint` changes to top-level LOBs.

**Inheritance is re-sync-by-divergence, not class extension.** The flow pins `Last synced: YYYY-MM-DD` once. Cold-boot scans every ancestor's `CHANGELOG.md` for entries newer than that date and surfaces deltas one-line-per-ancestor. Apply manually on greenlight. The inheritance link is reference, not lock — the Acid Test is the invariant, not file naming or stage counts.

**When to introduce an intermediate blueprint.** When ≥ 2 instance flows share a recognizable shape that diverges in stage names/ordering and would benefit from a canonical reference. The canonical exemplar (2026-04-28): wholesaling's three short-sale deals (Acacia, Rowlett, Biscayne) shared a short-sale shape, diverged in stage names, and prompted `_short-sale-deal-blueprint/` as their common parent.

## Absorption: the prose-to-flow modality

**Absorption is the act of converting an oversized static prose document into a catalog flow.** It is a recognized `/flow` modality alongside `audit`, `streamline`, `scaffold`, and `realize`.

**Trigger conditions for absorption:**
1. Source document is 500+ lines (overbearing as a static reference)
2. Document exhibits consistent repeated structure across sections (e.g., one concept per H3, all H3s follow same field pattern)
3. Units are independent — readers don't need to read the whole document first
4. Content maps cleanly to flow section shape (concept, techniques, executable spec, references)
5. Value of conversion: makes documentation executable (each entry is a step Claude can walk) and discoverable (`ls processes/` surfaces all options)

**The absorption motion (5 steps):**
1. **Identify the repeated unit** — what is the consistent structure? (e.g., H3 per blueprint in tome-of-geometry)
2. **Test: pipeline or catalog?** — Are units sequential with dependencies? → pipeline. Independent options? → catalog.
3. **Map source → step.md sections** — Concept block, stack, key techniques, build-instruction block (re-voiced from cold-start spec), inspired-by. Keep faithful to source; don't sanitize structure for aesthetics.
4. **Write init.md** — Catalog variant if appropriate (metadata table + selection guidance + blueprint ref + deviation notes if using non-standard numbering or shapes).
5. **Archive the original** — Leave a pointer stub at source path (~20 lines); move original to `<source-dir>/_archive/`.

**Shape decisions for absorbed catalogs:**
- **Numbering:** 2-digit for fixed-count ≤50 entries; 4-digit for unbounded or >100.
- **Elegance thesis / motivational field:** Absorbed into concept paragraph inline, not as a separate field (keeps step.md scannable).
- **CLAUDE.md:** Omit for catalog flows with no LOB-specific overrides beyond blueprint defaults.
- **Graduation gates:** Omit for independent catalog entries (only meaningful at pipeline stage boundaries).
- **Init.md flow diagram:** Catalog flows document as "selection model" in prose, not as Mermaid linear pipeline.

**How to apply:** When `/flow realize` encounters Shape 2 (Oversized Prose) with consistent internal structure, test absorb-ability before recommending decomposition into `docs/`. Ask: (a) are units independent? (b) does content map cleanly to step.md? (c) would `ls processes/` expose valuable options? If yes, absorb. If no, decompose via realize.

**Absorption as preparation for `/flow absorb` verb (future wave):** This wave documents the motion and shape decisions. A future wave will implement a `/flow absorb` verb that automates detection, mapping, and init.md generation. The verb lands on settled doctrine, not invented in isolation.

**Precedent:** K-EA2 wave (aquila) absorbed `tome-of-geometry/16-prototype-patterns.md` into `/flows/prototype-patterns/` (25 entries). Recipe and doctrine amendments in K-EA3 wave (aries).

## The cold-boot sequence (any flow, any reader)

```
1. ls <LOB>/                           → init.md is loudest
2. read init.md                        → mission + read order + current stage + tier
3. read CLAUDE.md                      → norms (cascades from portfolio)
4. read docs/cold-boot-brief.md        → state capsule
5. ls processes/                       → see the tree, locate current stage
6. read processes/<active>/<artifact>  → stage doctrine (in/out/owner/loopbacks)
7. ls docs/sessions/                   → next concrete work brief (if any)
```

**Walk-test target:** within 5 listings + 2 file reads, comprehension is complete. **Cold-boot timing target:** < 90 seconds from first `ls` to "I know what to do next."

## How a flow graduates

A stage stays at its current tier unless **all three** conditions bark simultaneously:

1. **Volume bark** — manual repetition has become humanly intractable.
2. **Audience bark** — the artifact has graduated past Tier 1 (roster or public uses it daily).
3. **Stakes bark** — compliance, legal, or data-integrity demands it.

Until all three apply: ship the file version and revisit when the bark gets louder. This is `thin-engine.md` doctrine applied to flow stages.

## The living blueprint mechanic

`_flow-blueprint/` is itself a flow. It eats its own dogfood. It maintains a `CHANGELOG.md`. Every active flow's `init.md` carries a `## Blueprint reference` footer with a `Last synced` date and a pointer to the CHANGELOG entry current at sync.

On **cold-boot AND after every flow run**, when an active flow's `Last synced` date lags behind a blueprint CHANGELOG entry, the reading Claude evaluates the divergence:

- **Meaningful divergence** (new archetype, new graduation-tier example, new section pattern per CHANGELOG header): surface to Dan — one line: *"Blueprint added X on YYYY-MM-DD; should this flow inherit?"* Dan greenlights or defers.
- **Minor divergence** (prose edit, spelling fix, solo-LOB experiment per CHANGELOG's own "not meaningful" definition): write a one-liner to `docs/backlog/blueprint-sync-candidates.md` (create if absent, append if present). Passive sink — no surface to Dan unless asked.

The feedback loop:

```
Active flow ships work
        ↓
AAR reveals friction or new generalizable pattern
        ↓
CHANGELOG entry added to _flow-blueprint/ (if pattern generalizes)
        ↓
Other flows see the divergence on next cold-boot
        ↓
Dan greenlights re-sync per flow, per change
        ↓
Flows that adopt the pattern update their Blueprint reference footer
```

## Why "flow" not "pipeline" or "process"

- **Pipeline** implies one direction, no feedback. Flows have AAR loops back into the blueprint.
- **Process** implies the work is the point. Flows are the *motion of an LOB* — the LOB is the work; the flow is how the LOB moves.
- **Workflow** is the genus; **flow** is what the family of process-shaped and content-shaped LOBs share. Workflow LOBs (`workflow-lob.md`) are a special case where the flow IS the entire deliverable.
